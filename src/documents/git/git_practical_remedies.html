Git practical remedies

A force-push avoidance guide for everyday Git work: sync before you push, keep work on feature branches, and abort messy rebases before they turn into risky history edits.

Core rule

The force push happened because a rebase conflict got stuck. Next time, resolve conflicts cleanly or abort and use a simpler merge approach instead of forcing through.

Never force push unless absolutely necessary. If force pushing becomes common, the workflow needs adjustment.

Practical remedies

  1. 1Always pull before pushing 

    git pull --rebase
git push

  2. 2Configure Git to rebase by default 

    git config --global pull.rebase true

  3. 3Use feature branches instead of master 

    git checkout -b feature/my-work
# work and commit
git push origin feature/my-work
# then merge via PR

  4. 4Sync cloud work before leaving 

    git pull --rebase
git push

  5. 5Stash locally before cloud work 

    git stash
# do cloud work
git pull --rebase
git push
# back locally
git stash pop

  6. 6Fetch to check remote changes first 

    git fetch
git log HEAD..origin/master
git pull --rebase
git push

    Use the log command to see what your local branch is behind before pushing.

Resolving Git conflicts cleanly

When rebase gets stuck

  • git rebase --continue hangs or shows no output.
  • Multiple conflict markers appear.
  • You are unsure which changes to keep.

Showcase: JSON conflict resolution

JSON files are sensitive to markers. A single <<<<<<< breaks the entire structure.

  1. Locate: git status shows both modified: index.json.
  2. Edit: Open index.json, find conflict blocks, and ensure the resulting JSON is valid (check commas and brackets).
  3. Stage: git add index.json.
  4. Proceed: git rebase --continue.

Abort if stuck

git rebase --abort
# returns you to the state before rebase started

Use regular merge instead

git pull
# resolve conflicts in files
git add <resolved-files>
git commit
git push

If you must rebase, do it step by step

git fetch
git rebase origin/master
# when conflict appears:
# edit the file
# remove conflict markers
git add <file>
git rebase --continue
# if it gets stuck again:
git rebase --abort

Conflict resolution best practices

Understand what you are merging

git log --oneline --graph --all

Check what may conflict

git fetch
git diff HEAD origin/master

Resolve manually

  • Open conflicted files in your editor.
  • Compare both versions: HEAD and incoming.
  • Choose the correct code or combine both.
  • Remove <<<<<<<, =======, and >>>>>>> markers completely.
  • Test the changes.

Verify before committing

git status
git diff --cached
# run tests if applicable
git commit

Abort when

  • There are too many conflicts to handle cleanly.
  • You are unsure which changes are correct.
  • The rebase has been running unusually long.
  • You want to get back to a working state.

Resolve when

  • Conflicts are few and clear.
  • You understand both sets of changes.
  • The changes are important to preserve correctly.

Alternative: use merge commits

Instead of rebasing, which rewrites local commit shape, use merge commits to preserve both histories. 

git config --global pull.rebase false
git pull
# creates a merge commit preserving both histories
# no force push needed

Prevention checklist

Before leaving cloud environment

git status
git pull --rebase
git push

Before starting local work

git fetch
git status
# check if you are behind

Use feature branches

git checkout -b feature/work
# work on feature branch
git push origin feature/work
# merge via PR when done
# master stays clean
