Git repository renaming

A safe, repeatable process for changing a repository name on GitHub and updating the local clone without losing remote history, breaking remotes, or leaving the repo in an inconsistent state.

safe remote + local github cli git remote update

What this process covers

  • Rename the repository on GitHub first.
  • Update the local `origin` remote to the new name.
  • Optionally rename the local folder to match the new repo name.
  • Verify the new remote before pushing again.

Process

1

Confirm current remote

Read `git remote -v` and identify the owner, repo, and URL style.

2

Rename on GitHub

Use the GitHub API or `gh api` to patch the repository name.

3

Update local remote

Set `origin` to the new repo URL and verify with `git ls-remote`.

4

Optionally rename folder

Rename the local directory only after the remote update is confirmed.

Workflow

  1. Check the current remote and make sure the repo is clean.
  2. Use a dry-run mode first to preview the owner, current name, and target name.
  3. Require an explicit confirmation flag before any GitHub rename.
  4. Update the local remote URL to the renamed repository.
  5. Verify the new remote still resolves before pushing anything else.

Recommended safety gates

  • Refuse detached HEAD state.
  • Refuse unsupported remote URL formats.
  • Refuse directory rename if the target path already exists.
  • Refuse a live rename without `--yes`.

Example cases

Case 1: SSH remote

Current remote looks like `git@github.com:owner/repo.git`.

scripts/rename_github_repo.sh new-name --yes

Case 2: Rename local folder too

Use this when you want the filesystem path to match the new repo name.

scripts/rename_github_repo.sh new-name --yes --rename-dir

Case 3: Preview only

Use dry-run first to verify the parsed owner and repo name.

scripts/rename_github_repo.sh new-name --dry-run

Case 4: HTTPS remote

The script accepts `https://github.com/owner/repo.git` too and preserves that style.

scripts/rename_github_repo.sh new-name --yes --remote origin

Variants and edge cases

  • Only the repository name changes. The owner stays the same unless you override it explicitly.
  • If your repo is part of an org transfer, update the owner separately and treat that as a distinct operation.
  • If CI uses full history or branch syncs, check out with `fetch-depth: 0` before running rename-related automation.
  • If the repo was renamed externally, update local remotes and verify before your next push.
