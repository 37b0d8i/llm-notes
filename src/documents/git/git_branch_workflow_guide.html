Git branch workflow

The pattern that keeps your code stable and team productive. Main stays safe. Features stay isolated. Changes get reviewed.

Core principle

Main stays stable. Feature branches are isolated. PRs enforce review. This pattern keeps your code safe and your team productive.

The four phases

1

Main is sacred

The main branch is your source of truth. It's production-ready code that's been reviewed and tested. Never commit directly to main — protect it so only merged PRs can update it.

This gives you confidence that main is always stable and deployable.

2

Feature branches = isolation

When you start work, create a feature branch off main:

$ git checkout main
$ git pull origin main
$ git checkout -b feature/auth

You work here and make commits. Meanwhile, team members work on their own branches (feature/payments, feature/dashboard) — all in parallel without stepping on each other's toes.

Key point: Feature branches are your sandbox. Make as many commits as you want. Rewrite history if you need to. It doesn't matter because it's not shared yet.

3

Pull requests = review gate

When your feature is ready, open a PR. This is not a merge yet — it's a proposal:

  1. Open PR from feature/auth → main
  2. Team reads, tests, and requests changes
  3. You update your feature branch (just git push — PR auto-updates)
  4. When approved: click merge and your commits fold into main

Why PRs matter:

  • Catches bugs before production
  • Shares knowledge across the team
  • Enforces a standard (every line reviewed by at least two pairs of eyes)
  • Creates a paper trail (the PR documents why decisions were made)
4

Merge back to main

When approved, the PR merges. Git replays your feature commits on top of main's current state:

main: c1 -- c2 -- c3 -- (merge: feature/auth commits)

Your feature branch can now be deleted — it's served its purpose. The commits are permanently recorded in main's history.

Real example: three-day feature

Day 1: Start auth feature
$ git checkout -b feature/auth
$ git add login.js
$ git commit -m "add login form"
$ git add auth-service.js
$ git commit -m "add authentication service"
$ git push origin feature/auth
Day 2: Open PR, get feedback
→ GitHub opens PR, team reviews
→ You make changes based on feedback
$ git add error-handling.js
$ git commit -m "add error handling"
$ git push origin feature/auth # PR auto-updates
Day 3: Approved & merged
→ PR merges automatically
→ Your commits now on main
→ Delete feature/auth (cleanup)

From day 1 to day 3, main was untouched. Other team members shipped their own features on separate branches. When your PR merged, it became part of main's history.

Why this pattern works

  • Stability: main always works. Hotfixes, deploys, and rollbacks are safe.
  • Parallelism: Two people on auth and payments simultaneously. No merge conflicts because they're on separate branches.
  • Clarity: PR history is your project diary. Why does this code exist? The PR answer is there.
  • Safety: You can reset or rewrite your feature branch without affecting main. Only reviewed code matters.
